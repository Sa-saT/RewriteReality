# 09. 土台選定の経緯（VFX / realtime 業界マップ）

なぜ Unity を選んだかの記録。判断材料として残す。

## 前提（このアプリの要件）

- **無料**で作りたい
- **Web に依存しない**ネイティブ単体アプリ
- メンテの止まった**個人アドオンに依存したくない**（保守性）
- リアルタイムでカメラを追従合成 → エフェクト → 複数出力（投影 / Syphon / NDI）

## VFX 業界の主流（オフライン vs リアルタイム）

### A. オフライン VFX（映画・CM のポスプロ）※今回の用途ではない
- **Houdini**（シミュレーション・プロシージャル）
- **Nuke**（コンポジット業界標準）
- **After Effects / DaVinci Resolve(Fusion)**（CM・MV）
- **Maya / Cinema 4D**（3D）
→ リアルタイムではないので「今この瞬間のカメラを埋め込む」用途に不向き。

### B. リアルタイム / ライブ映像・メディアアート ★このアプリの土俵

| ツール | 立ち位置 | 主流度 | 無料? |
|---|---|---|---|
| **TouchDesigner** | インスタレーション・ライブ・プロジェクションの事実上の標準 | ◎最主流 | 非商用無料(解像度制限)/商用有料 |
| **Notch** | コンサート・放送・XR ステージのハイエンド | ◎(大規模) | 高額 |
| **Unreal Engine** | バーチャルプロダクション(LED ウォール)の標準 | ◎(VP) | 無料(売上%) |
| **Unity** | インタラクティブ・インスタレーション・AR・一部 VJ | ○広く使用 | 無料(Personal) |
| **Max/MSP Jitter** | 音連動・実験系。Rhizomatiks が実際に使用 | ○ | 有料 |
| **openFrameworks / Processing / Cinder** | クリエイティブコーディングの伝統 | △ニッチ(TD に押され気味) | 無料 OSS |
| **Resolume** | クラブ VJ の定番(ループ素材ベース) | ◎(クラブ VJ) | 有料 |
| **disguise(d3) / Smode** | 巨大ショーのメディアサーバ/ショー制御 | ◎(超大規模) | 高額 |

## 判断

- 「準備映像にライブ映像をトラッキング合成してエフェクトして出力」という処理の
  **業界主流は TouchDesigner / Notch（ノードベース）**。ただし**無料条件と衝突**
  （TD は商用有料・非商用は制限、Notch は高額）。
- **無料 ＋ ネイティブ ＋ 業界で廃れていない**を全部満たす最良のバランスが **Unity**。
  realtime 業界で広く使われ、Keijiro 資産（Klak）で保守も無料で確保できる。
- **oF** は完全 OSS だが主流から外れつつあり、アドオン保守の弱さが懸念（→ `10` で別途検討）。

### 最終選択: **Unity 6 LTS（URP）＋ Klak ＋ VFX Graph ＋ OpenCvSharp（すべて無料）**

| 条件 | Unity の充足 |
|---|---|
| 無料 | ◎ Personal＋OSS のみ |
| Web 非依存ネイティブ | ◎ .app ビルド |
| 保守性 | ◎ Unity 本体＋Keijiro 現役保守。唯一 OpenCvSharp の native 導入に手間 |
| realtime 性能 | ○ 実用上十分（GC 配慮） |
| 情報量 | ◎ |

## 補足: 将来 TouchDesigner も併用しうる

本アプリは Syphon/NDI 出力を持つので、**合成・トラッキングは Unity、
最終のミックス/マッピングは TouchDesigner や Resolume** という現場連携も可能。
土台を Unity にしても他主流ツールと共存できる。
